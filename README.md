## UART ASIC Project (Sky130 UART Core)

### Overview

This project implements a simple, full‚Äëcustom UART (Universal Asynchronous Receiver/Transmitter) core targeted for ASIC fabrication on the **SkyWater SKY130** process.  
It contains:

- **UART transmitter and receiver RTL** (`uart_tx.v`, `uart_rx.v`) wrapped in a top‚Äëlevel core (`uart_top.v`)
- **Memory‚Äëmapped I/O (MMIO) wrapper** (`uart_mmio_wrapper.v`) suitable for integration into a small SoC
- **System‚Äëlevel testbench** (`uart_system_tb.v`) that models a CPU writing/reading MMIO registers
- **Example C driver** (`software/uart_driver.c`) showing how a CPU would send characters
- **ASIC configuration and results**:
  - `config.json` ‚Äì OpenLane/OpenROAD configuration targeting Sky130
  - `gds/uart_top.def` ‚Äì placed & routed DEF
  - `reports/` ‚Äì timing and power reports for the synthesized and routed design

The design is parameterized by `CLKS_PER_BIT` so you can match the UART baud rate to your system clock. In the provided top‚Äëlevel, `CLKS_PER_BIT = 87`, which corresponds to a baud rate of approximately 115200 bps at a 10 MHz clock.

### Repository Layout

- **`uart_tx.v`**: UART transmitter finite‚Äëstate machine.
- **`uart_rx.v`**: UART receiver finite‚Äëstate machine.
- **`uart_top.v`**: Combines transmitter and receiver into a single UART core.
- **`uart_mmio_wrapper.v`**: MMIO wrapper exposing a simple 32‚Äëbit bus interface around `uart_top`.
- **`uart_system_tb.v`**: Top‚Äëlevel testbench that instantiates `uart_mmio_wrapper` and emulates a CPU performing MMIO accesses.
- **`software/uart_driver.c`**: Example bare‚Äëmetal driver that writes characters to the UART via memory‚Äëmapped registers.
- **`config.json`**: Flow configuration (PDK, standard‚Äëcell library, clock constraints, and RTL file list) for running the digital ASIC flow (e.g. with OpenLane/OpenROAD).
- **`gds/uart_top.def`**: Example placed & routed DEF for `uart_top`.
- **`reports/`**:
  - `timing_min.rpt` ‚Äì minimum‚Äëdelay (hold) timing checks (slack MET).
  - `power.rpt` ‚Äì power estimation report (if generated by your flow).

### RTL Design Summary

#### `uart_tx.v` ‚Äì Transmitter

- Parameter: **`CLKS_PER_BIT`** ‚Äì number of clock cycles per UART bit.
- Inputs:
  - `clk`, `rst`
  - `start` ‚Äì pulse high to start sending the byte in `data_in`
  - `data_in[7:0]` ‚Äì data byte to transmit (LSB first)
- Outputs:
  - `tx` ‚Äì serial transmit line (idles high)
  - `tx_busy` ‚Äì asserted while a frame (start, 8 data bits, stop) is in progress
- Internal FSM states: `IDLE`, `START`, `DATA`, `STOP`.

#### `uart_rx.v` ‚Äì Receiver

- Parameter: **`CLKS_PER_BIT`** ‚Äì number of clock cycles per UART bit (must match transmitter).
- Inputs:
  - `clk`, `rst`
  - `rx` ‚Äì serial receive line
- Outputs:
  - `data_out[7:0]` ‚Äì received data byte
  - `rx_done` ‚Äì one‚Äëcycle pulse when a full byte has been received
- Internal FSM states: `IDLE`, `START`, `DATA`, `STOP`, with mid‚Äëbit sampling to tolerate skew.

#### `uart_top.v` ‚Äì Core Wrapper

- Instantiates `uart_tx` and `uart_rx` with a shared `CLKS_PER_BIT` parameter.
- Exposes:
  - `tx_start`, `tx_data_in[7:0]`, `tx_busy`
  - `uart_tx_out`, `uart_rx_in`
  - `rx_data_out[7:0]`, `rx_done`

### MMIO Interface (`uart_mmio_wrapper.v`)

`uart_mmio_wrapper` adapts `uart_top` to a basic 32‚Äëbit memory‚Äëmapped bus:

- **Inputs**
  - `clk`, `rst`
  - `addr[3:0]` ‚Äì 4‚Äëbit word address from the CPU
  - `wdata[31:0]` ‚Äì write data
  - `we` ‚Äì write enable (1 = write, 0 = read)
- **Outputs**
  - `rdata[31:0]` ‚Äì read data to CPU
  - `uart_tx`, `uart_rx` ‚Äì external serial pins

Address map (word‚Äëaddressed via `addr`):

- **`addr = 0x0`**
  - **Write**: pulses `tx_start` for one clock (transmit `wdata[7:0]`).
  - **Read**: returns `{31'b0, tx_busy}` to let the CPU poll TX status.
- **`addr = 0x4`**
  - **Read**: returns `{24'b0, rx_data}` to allow the CPU to read the last received byte (when `rx_done` is seen).

This simple interface can be connected to an APB/AHB‚Äëlite style bus using a small address decode shim in a full SoC.

### Example Software Driver (`software/uart_driver.c`)

The C file demonstrates a minimal way to drive the UART from a CPU with a memory‚Äëmapped interface:

- Base address: `UART_BASE = 0x40001000` (example SoC address).
- Data and status registers are accessed through volatile pointers at this base.
- Function:
  - `void uart_send_char(char c)`:
    - Polls the status register until TX is not busy.
    - Writes the character to the data register to trigger a transmit.

You can build on top of this to add:

- `uart_init()` ‚Äì to configure baud rate and parity (if additional registers are added).
- `uart_send_string(const char *s)` ‚Äì to send C strings.
- `uart_recv_char()` ‚Äì to block until a character is received (using the RX status).

> Note: The exact SoC address map and bus glue logic are platform‚Äëdependent; adjust `UART_BASE` and register offsets according to your SoC‚Äôs memory map.

### Simulation

You can simulate the complete system (CPU + MMIO wrapper + UART core) using `iverilog` and `vvp`.  
The provided testbench `uart_system_tb.v`:

- Generates a clock (`clk`) with 10 ns period (100 MHz / 2 toggling every 5 ns).
- Drives reset and MMIO bus signals (`addr`, `wdata`, `we`).
- Writes the ASCII character `'A'` to the UART data register.
- Polls the status register until TX is finished.
- Waits for `rx_done` and then reads back the received byte.
- Dumps waveforms to `system.vcd`.

#### Prerequisites

- `iverilog` (Icarus Verilog)
- `vvp` (runtime for Icarus Verilog)
- `gtkwave` (optional, for viewing waveforms)

#### Running the Testbench

From the project root:

```bash
# Compile
iverilog -g2012 -o uart_system_tb.out \
  uart_tx.v uart_rx.v uart_top.v uart_mmio_wrapper.v uart_system_tb.v

# Run simulation
vvp uart_system_tb.out

# View waveforms (optional)
gtkwave system.vcd &
```

You should see:

- Console messages from `$display` in the testbench.
- A complete transmit‚Äëand‚Äëreceive loopback on `uart_wire` in the waveform.

### ASIC Flow (OpenLane / OpenROAD)

The project is set up to run through a standard Sky130 digital ASIC flow using tools such as **OpenLane** / **OpenROAD**.

Key configuration is in `config.json`:

- **`PDK`**: `"sky130A"`
- **`STD_CELL_LIBRARY`**: `"sky130_fd_sc_hd"`
- **`DESIGN_NAME`**: `"uart_top"`
- **`VERILOG_FILES`**: lists `uart_tx.v`, `uart_rx.v`, and `uart_top.v`
- **`CLOCK_PORT`**: `"clk"`
- **`CLOCK_PERIOD`**: `10.0` ns (100 MHz target)
- **`FP_CORE_UTIL`**: floorplan utilization (e.g. 30%)
- **`RT_MAX_LAYER`**: `"met5"` (max routing layer)

#### Typical Flow (High‚ÄëLevel)

1. **Set up Sky130 and tools**  
   Install and configure a Sky130‚Äëenabled flow environment (e.g. `iic-osic-tools`, OpenLane2 Docker, or a local OpenROAD toolchain).

2. **Point the flow to this design directory**  
   Use the appropriate command for your flow to run synthesis, floorplanning, placement, CTS, routing, and sign‚Äëoff using `config.json` as the design configuration.

3. **Inspect results**
   - Check `reports/` for:
     - **Timing**: e.g. `reports/timing_min.rpt` shows hold checks with positive slack (MET).
     - **Power**: e.g. `reports/power.rpt` (if generated).
   - Inspect `gds/uart_top.def` (and GDS if produced) using your preferred layout viewer (e.g. KLayout, Magic).

> The exact flow invocation commands depend on your environment (OpenLane version, Docker vs local). Use this directory as the design root and ensure the PDK path and `config.json` are correctly referenced in your flow scripts.

### Customization

- **Baud Rate / Clock**  
  Adjust the `CLKS_PER_BIT` parameter in `uart_top.v` (and any instantiations) to match your system clock and desired baud rate:
  \[
  \text{CLKS\_PER\_BIT} \approx \frac{f_\text{clk}}{\text{baud}}
  \]

- **Additional Registers**  
  Extend `uart_mmio_wrapper.v` to:
  - Expose RX status bits (e.g. `rx_done`, error flags).
  - Support configuration registers (baud divisor, parity, stop bits).

- **Integration into a SoC**  
  Connect `uart_mmio_wrapper` to your SoC bus, mapping `addr`, `wdata`, `we`, and `rdata` to the appropriate address range and bus protocol.
